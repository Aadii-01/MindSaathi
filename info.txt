1. PROJECT OVERVIEW
================================================================================

Purpose:
Smriti is a full-stack web application designed for cognitive assessment and 
mental performance tracking. It provides a comprehensive platform for users to 
evaluate and improve their cognitive abilities through scientifically-designed 
tests, track performance trends, and receive personalized insights.

Key Features:
- User authentication (Signup, Login, Admin Login)
- 6+ cognitive assessment tests with real-time scoring
- Performance tracking with visual charts and trend analysis
- Bilingual support (English and Hindi)
- Personalized feedback collection system
- Mind State Analysis (AI-powered emotional assessment)
- Responsive, mobile-first UI design
- Industry-standard glass morphism design aesthetic

Use Cases:
- Cognitive self-assessment for individuals
- Research and data collection for institutions
- Employee cognitive performance monitoring
- Educational cognitive training platforms
- Mental health assessment tools

A) NUMBER MEMORY TEST
   =====================
   File: frontend/src/pages/tests/NumberMemoryTest.jsx
   Route: /NumberMemoryTest
   Purpose: Assess working memory and digit recall ability
   
   Configuration:
   - Total Questions: 10
   - Time Limit per Question: 25 seconds
   - Initial Digit Length: 3 digits
   - Maximum Digit Length: 10 digits
   - Display Duration: 3-5 seconds (random)
   
   Test Flow:
   1. SHOW PHASE: Display random number string (3+ digits)
      - Duration: 3-5 seconds
      - User sees the number clearly
   2. HIDE PHASE: Hide the number
      - User must wait and memorize
      - Preparation time: 1-2 seconds
   3. INPUT PHASE: User enters the number
      - Input field becomes active
      - User has up to 25 seconds to type answer
      - Time countdown displayed
   
   Difficulty Progression:
   - Question 1-3: 3-4 digits (Easy)
   - Question 4-6: 5-6 digits (Medium)
   - Question 7-8: 7-8 digits (Hard)
   - Question 9-10: 9-10 digits (Very Hard)
   
   Scoring Algorithm:
   - Correct Answer: +1 point
   - Wrong/Timeout: 0 points
   - Final Score: Number of correct answers (0-10)
   - Efficiency Score: Score * 100 / Total Questions
   
   Technical Implementation:
   - generateNumberString(): Creates random digit sequence
   - useRef() for DOM timing and state management
   - setTimeout() for phase transitions
   - Phase-based UI rendering (show/hide/input)
   
   State Variables:
   - phase: "show" | "hide" | "input"
   - displayValue: Current number being displayed
   - answer: User's typed input
   - attempt: Current question number (1-10)
   - correctCount: Total correct answers
   - wrongCount: Total wrong answers
   - digitLength: Current difficulty level
   
   Report Data Sent to Backend:
   {
     score: number (0-10),
     total: 10,
     test_type: "number_memory",
     username: string,
     attempts: number,
     correctCount: number,
     wrongCount: number,
     digit_span_max: number
   }
   
   Performance Metrics:
   - Accuracy: correctCount / 10 * 100
   - Speed: Total time taken
   - Digit span achieved: Maximum correct digit length
   - Consistency: Standard deviation across attempts

B) FIND THE TARGET TEST
   ======================
   File: frontend/src/pages/tests/FindTargetTest.jsx
   Route: /FindTargetTest
   Purpose: Assess visual search speed and attention span
   
   Configuration:
   - Total Questions: 10
   - Time Limit per Question: 20 seconds
   - Grid Size Min: 5x5 (25 cells)
   - Grid Size Max: 9x9 (81 cells)
   - Target Character: "A"
   - Non-Target Characters: ["B", "C", "D", "E", "F", "G", "H"]
   
   Test Flow:
   1. Display grid with random characters
   2. User searches for target character "A"
   3. User clicks on the cell containing "A"
   4. Feedback: Correct/Wrong/Too Late
   5. Move to next question with increased grid size
   
   Difficulty Progression:
   - Question 1: 5x5 grid (25 cells) - Easy
   - Question 2-3: 5-6x5-6 grid
   - Question 4-6: 6-7x6-7 grid - Medium
   - Question 7-8: 7-8x7-8 grid
   - Question 9-10: 8-9x8-9 grid - Very Hard
   
   Scoring Algorithm:
   - Correct First Click: +1 point
   - Wrong Clicks: Tracked but no penalty
   - Timeout: 0 points for question
   - Final Score: Total correct (0-10)
   
   Technical Implementation:
   - makeGrid(): Generates grid with target and non-targets
   - CSS Grid layout for responsive cell arrangement
   - Click event handling on grid cells
   - Time countdown with visual timer
   - Real-time feedback display
   
   State Variables:
   - gridSize: Current grid dimensions (5-9)
   - grid: Array of characters in grid
   - targetIndex: Position of target character
   - selectedWrongCount: Number of wrong clicks
   - missedTargets: Questions with no correct answer
   - correctSelections: Total correct answers
   - timeLeft: Remaining time for current question
   - phase: "playing" | "finished"
   
   Report Data Sent:
   {
     score: number (0-10),
     total: 10,
     test_type: "find_target",
     username: string,
     correctCount: number,
     wrongCount: number
   }
   
   Performance Metrics:
   - Visual Search Speed: Average time per question
   - Accuracy: Correct / Total
   - Distraction Rate: Wrong clicks / Total clicks
   - Consistency: Performance trend across difficulty levels

C) PATTERN COMPLETION TEST
   ========================
   File: frontend/src/pages/tests/PatternCompletionTest.jsx
   Route: /PatternCompletionTest
   Purpose: Assess logical reasoning and pattern recognition
   
   Configuration:
   - Total Questions: 10
   - Time Limit per Question: 25 seconds
   - Pattern Types: 3 types (rotated across questions)
   - Difficulty Levels: Easy (3 Q), Medium (3 Q), Hard (4 Q)
   
   Pattern Types:
   1. NUMBER SEQUENCE
      - Arithmetic progression with varying steps
      - Easy: Simple +1, +2, +3 patterns
      - Medium: +2, +3, +4 patterns
      - Hard: +3, +4, +5 patterns with complex offsets
      Example: 2, 5, 8, 11, ? → Answer: 14
   
   2. SHAPE PATTERN
      - Geometric transformations and rotations
      - Sequences of shapes changing size/rotation
      - Visual pattern recognition required
      Example: Circle → Square → Triangle → ?
   
   3. SYMBOL SEQUENCE
      - Abstract symbol patterns
      - Mathematical or logical progression
      - Requires deductive reasoning
      Example: ◇, ◇◇, ◇◇◇, ?
   
   Scoring Algorithm:
   - Correct Answer: +1 point
   - Wrong/Timeout: 0 points
   - Final Score: Total correct (0-10)
   
   Technical Implementation:
   - buildPatternQuestion(): Generates pattern based on type/difficulty
   - Deterministic generation using seed (for consistency)
   - Multiple choice options (4-5 possible answers)
   - Difficulty scaling affects pattern complexity
   
   State Variables:
   - questionIndex: Current question (0-9)
   - patterns: Array of pattern objects
   - selectedAnswer: User's choice
   - correctCount: Correct answers so far
   - wrongCount: Wrong answers so far
   - timeLeft: Remaining time
   - phase: "playing" | "finished"
   
   Report Data:
   {
     score: number (0-10),
     total: 10,
     test_type: "pattern_completion",
     username: string,
     correctCount: number,
     wrongCount: number
   }
   
   Performance Metrics:
   - Logical Reasoning Score: Correctness
   - Pattern Recognition Ability: Consistency across types
   - Difficulty Adaptability: Performance by difficulty level
   - Processing Speed: Average time per question

D) STROOP TEST
   ==============
   File: frontend/src/pages/tests/StroopTest.jsx
   Route: /StroopTest
   Purpose: Assess cognitive control and selective attention
   
   Configuration:
   - Total Questions: 10
   - Time Limit per Question: 5 seconds
   - Color Options: RED, GREEN, BLUE, YELLOW
   - Conflict Ratio: 70% conflict, 30% matching
   
   Test Mechanism:
   - Display a COLOR WORD (e.g., "RED")
   - But display it in a DIFFERENT COLOR (e.g., Blue)
   - User must click the BUTTON matching the TEXT COLOR
   - NOT the word meaning, but the actual color
   
   Example Question:
   Word: "RED" (written in Blue text)
   Buttons: [RED] [GREEN] [BLUE] [YELLOW]
   Correct Answer: BLUE (the text color)
   
   Difficulty:
   - 70% Conflict: Word and color don't match → Cognitive load
   - 30% Matching: Word and color match → Easier, but unpredictable
   
   Scoring Algorithm:
   - Correct Answer: +1 point
   - Wrong Answer: 0 points
   - Timeout: 0 points
   - Final Score: Total correct (0-10)
   
   Technical Implementation:
   - generateStroopQuestions(): Creates question set
   - Conflict ratio control (70/30)
   - Button click event handlers
   - Real-time feedback (Correct/Wrong)
   - Color state management using COLOR_MAP
   
   State Variables:
   - questions: Array of stroop question objects
   - questionIndex: Current question index
   - selectedButton: User's button click
   - correctCount: Total correct
   - wrongCount: Total wrong
   - timeLeft: Countdown timer
   - phase: "playing" | "finished"
   
   Psychological Basis:
   - Tests ability to ignore conflicting information
   - Measures executive function
   - Reveals cognitive control and decision-making speed
   - Often used in neurocognitive assessments
   
   Report Data:
   {
     score: number (0-10),
     total: 10,
     test_type: "stroop",
     username: string,
     correctCount: number,
     wrongCount: number
   }
   
   Performance Metrics:
   - Cognitive Control: Accuracy despite interference
   - Attention Selective: Filtering irrelevant information
   - Decision Speed: Response time under conflict
   - Consistency: Performance across conflict/matching trials

Base URL: http://127.0.0.1:8000/api

API ENDPOINTS DOCUMENTATION:

A) AUTHENTICATION ENDPOINTS

1. POST /api/register/
   Purpose: User registration and account creation
   Frontend Implementation: Signup.jsx
   Request Body:
   {
     "username": "string (unique)",
     "email": "string (unique, valid email)",
     "password": "string (min 6 chars)",
     "full_name": "string (optional)"
   }
   Response (Success - 201):
   {
     "message": "User registered successfully",
     "token": "auth_token_string",
     "jwt": "jwt_token_string",
     "user": {
       "id": number,
       "username": "string",
       "email": "string",
       "full_name": "string",
       "is_admin": false
     }
   }
   Response (Error - 400):
   {
     "error": "Username already exists" | "Invalid email" | "Password too short"
   }
   Error Handling:
   - Username validation (3-30 chars, alphanumeric + underscore)
   - Email format validation
   - Password strength requirements
   - Duplicate username/email checking

2. POST /api/login/
   Purpose: User authentication
   Frontend Implementation: Login.jsx
   Request Body:
   {
     "username": "string",
     "password": "string"
   }
   Response (Success - 200):
   {
     "message": "Login successful",
     "token": "auth_token_string",
     "jwt": "jwt_token_string",
     "user": {
       "username": "string",
       "email": "string",
       "is_admin": false
     }
   }
   Response (Error - 401):
   {
     "error": "Invalid credentials" | "User not found"
   }
   Frontend Storage:
   - localStorage.setItem("jwt", response.jwt)
   - localStorage.setItem("user", JSON.stringify(response.user))
   - localStorage.setItem("user_token", response.token)

3. POST /api/admin-login/
   Purpose: Admin panel authentication
   Frontend Implementation: AdminLogin.jsx
   Request Body:
   {
     "username": "admin",

   Response (Success - 200):

     "message": "Admin login successful",

     "jwt": "admin_jwt_token",
     "user": {
       "username": "admin",

6. FRONTEND-BACKEND INTEGRATION FLOW
================================================================================

A) USER AUTHENTICATION FLOW

Step 1: User Signup
   Frontend (Signup.jsx)
     └─→ User fills form (username, email, password)
         └─→ Client-side validation
             └─→ POST /api/register/ with credentials
   
   Backend (views.py::register)
     └─→ Validate input (username unique, email format, password strength)
         └─→ Hash password with SHA-256
             └─→ Create user object
                 └─→ Generate JWT token (HS256)
                     └─→ Return token + user data
   
   Frontend
     └─→ Receive response
         └─→ Store JWT: localStorage.setItem("jwt", token)
             └─→ Store User: localStorage.setItem("user", JSON.stringify(user))
                 └─→ Redirect to /landing (dashboard)

Step 2: User Login
   Frontend (Login.jsx)
     └─→ User enters credentials
         └─→ POST /api/login/ with username + password
   
   Backend (views.py::login)
     └─→ Query user by username
         └─→ Verify password hash
             └─→ Generate JWT token
                 └─→ Return token + user data
   
   Frontend
     └─→ Store JWT and user data
         └─→ Redirect to /landing

Step 3: Authenticated Requests
   Frontend (Any component)
     └─→ Need to make authenticated API call
         └─→ Read JWT from localStorage
             └─→ Fetch API with Authorization header
                 └─→ fetch(URL, {
                      headers: {
                        "Authorization": "Bearer " + jwt
                      }
                    })
   
   Backend
     └─→ Middleware extracts bearer token
         └─→ Verify JWT signature (HS256 with SECRET_KEY)
             └─→ Decode token to get user claims
                 └─→ Attach user info to request
                     └─→ Process request

B) TEST COMPLETION AND RESULT SUBMISSION FLOW

Step 1: Test Execution
   Frontend (NumberMemoryTest.jsx, etc.)
     └─→ Initialize test with TEST_CONFIG
         └─→ Display questions in sequence
             └─→ Track user responses and timing
                 └─→ Calculate score
                     └─→ User completes all questions
                         └─→ Compile result object

Step 2: Result Submission
   Frontend (Result.jsx or test component)
     └─→ Navigate to Result page
         └─→ Prepare result payload:
             {
               score: number,
               total: number,
               test_type: "string",
               username: localStorage.user.username,
               correctCount: number,
               wrongCount: number,
               totalTimeSeconds: number
             }
         └─→ POST /api/result/ with result data
   
   Backend (views.py::save_result)
     └─→ Receive result object
         └─→ Generate unique ID
             └─→ Add timestamp
                 └─→ Save to JSON database
                     └─→ (Optional) Create database record
                         └─→ Return success response

Step 3: Result Retrieval
   Frontend (TestScores.jsx, ProfilePage.jsx)
     └─→ Component mounted
         └─→ GET /api/results/me/ with JWT auth
   
   Backend
     └─→ Verify JWT
         └─→ Extract username from claims
             └─→ Query all results for user
                 └─→ Return results array

9. SECURITY IMPLEMENTATION
================================================================================

A) PASSWORD SECURITY
   - Hash Algorithm: SHA-256 with salt
   - Salt: "smriti_salt_2025" (constant, for demo)
   - Storage: Hashed passwords only, original never stored
   - Verification: Compare hashes, not plaintext

B) TOKEN SECURITY
   - Token Type: JWT (JSON Web Tokens)
   - Signing Algorithm: HS256 (HMAC SHA-256)
   - Secret Key: Django SECRET_KEY from settings
   - Expiration: 24 hours after issue
   - Storage: localStorage (client-side)
   
   Token Structure:
   [Header].[Payload].[Signature]
   Where:
   - Header: Algorithm info (HS256, JWT)
   - Payload: User claims (id, username, exp, iat)
   - Signature: HMAC(SECRET_KEY, Header.Payload)

C) CORS SECURITY
   - Middleware: django-cors-headers
   - Allowed Origins: http://127.0.0.1:3000, http://localhost:3000
   - Allowed Methods: GET, POST, PUT, DELETE, OPTIONS
   - Allowed Headers: Content-Type, Authorization
   - Credentials: Included (cookies, auth headers)

D) AUTHENTICATION FLOW
   1. Frontend sends credentials to backend
   2. Backend validates and creates JWT
   3. Frontend stores JWT in localStorage
   4. Frontend includes JWT in all authenticated requests
   5. Backend verifies JWT signature before processing
   6. Backend extracts user info from verified JWT
   7. Request processed with user context

E) AUTHORIZATION
   - Admin endpoints check is_admin flag in JWT
   - User-specific endpoints verify username matches JWT subject
   - Feedback requires either authentication or anonymous submission

13. TROUBLESHOOTING & COMMON ISSUES
================================================================================

Frontend Issues:
1. "Can't reach backend"
   - Check: Is Django server running on port 8000?
   - Solution: python manage.py runserver
   
2. "CORS error"
   - Issue: Browser blocking cross-origin request
   - Solution: Verify CORS headers in Django settings
   
3. "Tests not displaying"
   - Issue: Async loading or state management
   - Solution: Check browser console for errors, verify API response
   
4. "localStorage not persisting"
   - Issue: Browser privacy mode or storage disabled
   - Solution: Check browser settings, use DevTools

Backend Issues:
1. "Database locked"
   - Issue: Multiple processes accessing SQLite
   - Solution: Use PostgreSQL for production, close other connections
   
2. "Token verification failed"
   - Issue: Invalid JWT or expired token
   - Solution: Check SECRET_KEY, token expiration, signature
   
3. "Media files not serving"
   - Issue: MEDIA_URL or MEDIA_ROOT misconfigured
   - Solution: Verify paths in settings.py
   
4. "CORS preflight fails"
   - Issue: OPTIONS request not allowed
   - Solution: Ensure corsheaders middleware is installed and configured
